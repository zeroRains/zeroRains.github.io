发表于2022-11-17|更新于2022-11-28|分布式
|阅读量:

基于Spark分布式计算系统的大规模分布式矩阵乘法的实现与评测

随着大数据时代的来临数据的规模和潜在价值呈现爆炸式的增长，为了充分利用和分析大数据，诸多企业和科学研究领域需要为各自的数据自行设计分析算法，而这些分析算法通常可以通过线性代数符号表示出来，其中数据就成为了矩阵，而数据分析操作就由矩阵计算（尤其是是矩阵乘法操作）组成。在大数据时代，数据大都存储于分布式环境系统环境中（如Hadoop的HDFS），操作这些数据的方式与在常规计算机上操作文件数据的方式不同，同时由于这些存储与分布式矩阵的数据通常是TB级的，想要像常规计算机一样将其读入内存并进行计算和分析是不现实的。为此，需要在分布式系统环境中加载数据，但由于不能像常规计算机一般操作文件，就需要使用分布式计算系统进行操作（如MapReduce，Spark，Flink等）。而分布式矩阵乘法的目的就在于充分利用分布式集群的性能，对TB级数据并发的进行矩阵乘法操作。

本课题将基于Spark分布式计算系统进行分布式矩阵乘法平台的开发，将多种分布式矩阵乘法算法集成到本课题的分布式矩阵乘法平台中，并针对不同的分布式矩阵乘法算法，在现实和合成的数据集中进行性能和开销的评测，分析各种分布式矩阵计算方法的使用优劣。

本课题的难点在于：

  1. 分布式环境下，矩阵的组织形式。与常规计算机直接将矩阵变量完全存入内存中不同，在分布式环境下，需要对矩阵的组织进行分块存储，并读取到集群内的不同节点中。由于分布式矩阵乘法是要对分布式矩阵进行计算，因此矩阵组织形式的优劣直接影响分布式矩阵计算的性能。
  2. 分布式矩阵乘法的通信开销。分布式矩阵乘法在本质仍然是元素级别的矩阵乘法操作，无论使用什么方法矩阵乘法的总执行次数都不会发生改变。但对于大规模数据进行矩阵乘法操作，只有借助分布式计算系统才能实现，如果使用常规的计算机进行处理，通常都会有内存溢出的问题。由于分块矩阵数据是在不同的分布式节点中进行存储的，当需要不同节点的矩阵数据进行计算的时候，这通常需要进行网络通信进行数据传输。由于分块矩阵的数量很大并且矩阵乘法计算过程中也会涉及到多个矩阵数据块的通信，因此分布式矩阵乘法计算的优化目标就在于尽可能地减少网络间的通信开销。
  3. 分布式矩阵乘法的评测方式。在实现分布式矩阵乘法之后，需要对分布式矩阵乘法算法进行评测。本课题使用现实数据集和合成数据集，对于合成数据集可以借助矩阵的组织形式进行直接生成，但是对于现实数据集，如何将其转化成本课题的平台支持的矩阵组织形式是一个需要考虑的问题，同时如何利用本平台提供的矩阵乘法方式实现现实数据集对应的任务也是一个需要考虑的问题。
  4. （下面的是可能实现，但是也可能实现不了的）
  5. 利用硬件对分布式矩阵计算进行加速。本课题期望能够借用GPU硬件设备对节点中的矩阵计算进行加速，但是通常情况下GPU的内存容量是比节点的内存容量小很多的，如何对节点中的数据矩阵进行重新组织，使其可以在GPU硬件上运行也是本课题的一个难点。
  6. 矩阵乘法链优化。在真实的数据分析应用中，不能只使用两个矩阵进行乘法就达到目标，他们通常由多个矩阵的矩阵相乘得到最终的结果。这就形成了一个矩阵乘法链，在不改变乘法链顺序的情况下，按照不同顺序的矩阵相乘得到的计算结果相同，但是计算过程中产生的中间数据和操作次数是不相同的，比如对于矩阵A，B和向量v，原本的矩阵乘法执行为A x B x v，如果将其执行顺序修改为A x (B x v)，其产生的中间变量的大小就会减小，因为如果按照原来的执行方式AxB之后得到一个矩阵，再和v相乘得到的结果就是一个矩阵-向量乘法。通常矩阵-矩阵乘法的开销都是比矩阵-向量乘法的开销大的。如果是后面的执行方式执行过程就会变成两个矩阵-向量乘法，从而减小了计算开销。
